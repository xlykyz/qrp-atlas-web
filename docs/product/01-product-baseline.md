# 产品定位与用户工作流

> 状态：阶段一 SSOT
> 基准日期：2026-07-16

## 1. 产品定位

QRP Atlas Web 是面向个人专业交易研究者的桌面研究终端。它不是通用后台、数据库浏览器、行情门户或聊天产品，而是把“发现问题—形成解释—记录判断—验证策略—复盘结果—检查数据可信度”串成连续工作过程的工作台。

一句话价值：**让用户在同一上下文中看清今天发生了什么、为什么值得研究、已有证据是什么、策略是否经得起验证，以及数据是否足够可信。**

## 2. 目标用户与使用环境

核心用户是能理解行情、指标、因子和回测口径的个人研究者，主要在桌面端进行每日盘后工作，也会在交易日内快速查看状态。典型特征：

- 同时追踪市场、题材、个股、研究记录和策略实验；
- 关心数据日期、可用时间、计算版本和异常，而不只关心数值；
- 需要从市场现象下钻到标的、证据、交易和回测结果；
- 需要重复执行、比较与复现，而不是一次性看图；
- Agent 只在当前研究上下文中辅助检索、解释与生成提案。

## 3. 产品原则

1. **当前问题优先**：第一屏先回答用户进入页面时最可能的问题，再提供完整数据。
2. **证据与时间常驻**：数据日期、更新时间、来源、版本、新鲜度、警告与口径不藏在帮助页。
3. **对象与过程连贯**：市场日、股票、题材、策略、任务、运行结果都具有稳定 URL；跨页面保持对象上下文。
4. **事实与推断分离**：后端事实、前端展示转换、研究笔记、Agent 推断分别标识。
5. **异常不静默**：loading、empty、error、stale、partial、success 都有明确界面；禁止失败后偷偷切 Mock。
6. **高密度而非拥挤**：用层级、对齐、分组、冻结和渐进展开承载密度，不用巨型卡片和空白制造“高级感”。
7. **研究终端而非管理后台**：导航围绕任务，不围绕表名、路由模块或 CRUD。
8. **A 股语义一致**：红涨绿跌；警告、失败、生命周期、证据状态使用独立语义，避免颜色混用。

## 4. 核心用户任务（JTBD）

### 4.1 每日启动与今日决策

当我开始当天研究时，我需要快速确认最新交易日、市场广度、极端状态、我的未完成研究、运行中的回测和数据异常，以决定今天先处理什么。

完成标准：5 分钟内形成有排序的工作清单，并能从每个异常直接进入对应对象。

### 4.2 市场复盘

当交易日结束后，我需要按日期复盘市场结构、涨跌分布、强弱持续性、极端个股、关键指数和我的判断记录，以便形成可回看的日度结论。

完成标准：市场事实、主观阶段判断、证据和交易行为处于同一交易日上下文。

### 4.3 个股研究

当一个标的值得研究时，我需要在指定 as-of 日期查看价格行为、基础资料、相关事件/研报/调研、我的阶段判断和交易记录，并把结论保存为研究记录。

完成标准：股票、日期、复权、图表窗口和详情 tab 可通过 URL 恢复。

### 4.4 动态产业与题材研究

当市场出现新叙事或题材状态变化时，我需要理解催化、产业链影响、成员证据、市场反馈、生命周期和争议，并将需要人工判断的关系进入审核过程。

完成标准：每个关键主张可回到证据；题材、产业链、公司关系与派生评分明确分层。后端能力未落地前仅建设契约适配和明确不可用状态。

### 4.5 研究记录与知识回放

当我形成判断时，我需要把笔记绑定到市场日、股票、题材、策略或回测运行，并能按时间、对象和标签回看，而不是把记录散落在各页面本地状态中。

完成标准：研究记录有稳定对象引用、时间、状态和后续动作。当前后端缺正式通用 research-note API，列为接口缺口。

### 4.6 指标、因子与策略研究

当我评估一个研究想法时，我需要先理解指标/因子的定义、频率、范围、输入和策略依赖，再查看策略版本、参数 schema、适用股票池和已有关联运行。

完成标准：前端不维护平行策略清单；目录和参数默认值来自后端 catalog/schema。

### 4.7 回测黄金工作流

```text
选择策略与版本
→ 阅读适用范围和依赖
→ 配置参数、股票池、日期、基准、仓位、成本和执行时点
→ 创建真实任务
→ 查看任务状态与错误
→ 打开标准结果
→ 查看指标、净值/回撤、基准、交易、跳过原因、成本、暴露、诊断和可复现性
→ 选择多个运行比较
→ 必要时从锁定快照重放
```

完成标准：所有正式数据来自 `qrp-atlas` API；任务和运行由 owner 隔离；URL 可直接打开任务、运行 tab 和比较集合。

### 4.8 数据运行与系统诊断

当结果异常或数据不新鲜时，我需要知道 API、数据表覆盖、最近日期、任务失败、缺失 artifact 和版本差异，以判断是市场事实、研究逻辑还是系统问题。

完成标准：系统页给出面向研究者的诊断，不把 SQL 控制台和原始表浏览放在一级导航。

## 5. 端到端工作流

### 每日主链

```text
今日工作台
→ 处理数据/任务异常
→ 打开市场复盘并锁定交易日
→ 下钻题材或个股
→ 写研究记录/更新判断
→ 发起策略验证或回测
→ 比较结果并记录结论
→ 回到今日工作台确认待办清零
```

### 研究验证链

```text
对象详情中的现象/证据
→ 打开相关指标或策略
→ 以当前对象上下文预填实验
→ 创建任务
→ 运行结果与诊断
→ 比较基线
→ 将结论写回研究记录
```

## 6. 明确非目标

- 不提供实盘自动交易、荐股或收益承诺；
- 不在浏览器复制因子、成交、PIT、权限和绩效算法；
- 不把 DuckDB/PostgreSQL、开发 SQL 或表浏览作为主产品；
- 不把 Agent 做成脱离当前对象的独立聊天首页；
- 不将 `qrp-global-atlas` 的 Cesium 三维场景并入主 Web；
- 不在后端缺接口时用永久 Mock 伪装完整业务能力；
- 不同时长期维护两套正式主前端。

## 7. 产品成功标准

- 每日核心状态在首屏可读且可下钻；
- 关键对象和筛选可由 URL 恢复；
- 黄金回测流程真实闭环、可重试、可比较、可复现；
- 数据时间、新鲜度、版本和警告在关键页面可见；
- 页面组件保持组合职责，业务域可独立扩展；
- 旧前端必要能力有明确覆盖、废弃或延期结论。
